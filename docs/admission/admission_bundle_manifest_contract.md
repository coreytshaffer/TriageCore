# Admission Bundle Manifest Contract

## Purpose

`bundle_manifest.json` is the smallest machine-readable contract inside the admission review bundle generated by `tc admission bundle --from-json <path> --out-dir <dir>`.

It exists to describe the review artifact contents without granting execution authority, implying admission approval, or claiming that runtime execution occurred.

## Current Manifest Shape

The current implementation writes this deterministic manifest shape:

```json
{
  "bundle_type": "admission_review",
  "execution_authority": false,
  "source_evidence": "admission_evidence.json",
  "rendered_review": "admission_review.md"
}
```

Only these fields are documented here because they are the fields currently emitted by the implementation.

## Field Contract

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `bundle_type` | `str` | Yes | Identifies the bundle as an admission review artifact. The current value is `admission_review`. |
| `execution_authority` | `bool` | Yes | Declares whether the bundle grants execution authority. The current contract requires `false`. |
| `source_evidence` | `str` | Yes | Relative filename for the copied admission evidence JSON inside the bundle. The current value is `admission_evidence.json`. |
| `rendered_review` | `str` | Yes | Relative filename for the rendered Markdown review artifact inside the bundle. The current value is `admission_review.md`. |

## Review-Only Boundary

The manifest must preserve the review-only boundary. In current terms, that means:

- `execution_authority` must remain `false`.
- The manifest does not grant execution authority.
- The manifest does not prove admission approval.
- The manifest does not prove runtime safety.
- The manifest does not prove execution occurred.
- The manifest does not bypass human review.

## Forbidden Manifest Semantics

The manifest should not introduce fields or meanings that blur review into approval or execution. Examples of risky field names or semantics include:

- `approved`
- `execution_granted`
- `admission_approved`

Those names suggest authority or approval state rather than review packaging and should be treated as contract drift unless a separate reviewed change explicitly redefines the manifest.

## Relationship To Bundle Output

The manifest is one part of the review bundle written to the explicit output directory:

```text
<out-dir>/
  admission_review.md
  admission_evidence.json
  bundle_manifest.json
```

The manifest must remain consistent with those filenames and with the broader admission review bundle goal: an operator-facing artifact path that helps review without smuggling in execution power.
