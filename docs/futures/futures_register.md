# Futures Register

This document tracks aspirational features, theoretical architectures, and future ideas for TriageCore that are not yet approved for implementation. 

**IMPORTANT GOVERNANCE RULE**: An item in the Futures Register **does not authorize code changes**. It may inspire future architecture, but any implementation must first be promoted to a formal Change Request (CR) and receive explicit human approval.

*Note: Phase 14 (Local Compute Fabric) currently remains in `docs/backlog.md`. In the future, Phase 14 may be split via a Change Request into actionable backlog items and aspirational Futures Register items.*

## Promotion Criteria
A Futures Register item can be promoted to a Change Request when:
1. The underlying technology or dependency becomes available and stable.
2. The core system architecture has matured enough to support the feature without compromising existing boundaries or the Self-Observing Harness doctrine.
3. A clear, bounded scope for implementation is defined.
4. There is human consensus to pursue the feature.

## Retirement Criteria
A Futures Register item should be retired (removed or marked as closed) when:
1. It contradicts a ratified Architectural Decision Record (ADR) or core doctrine.
2. The problem it intended to solve is no longer relevant due to other systemic changes.
3. Feasibility analysis determines it is technically or operationally impractical.
